## 文档目的

这一篇把当前还未完全解决的问题，以及下一步最值得继续推进的方向整理成统一清单。

它不是长期产品 roadmap，而是更偏当前工程执行层的开放问题列表。

## 当前阶段判断

目前主链路已经可用，但以下问题仍值得继续处理：

- 连接体验问题
- transport 和权限边界问题
- 少量非阻断日志污染
- 一些运行时语义仍需继续保持清晰

## 一类：连接与权限体验

### USB 权限交互还不够顺滑

当前典型体验问题包括：

- 插入设备后不会立刻弹权限
- 用户点击连接时，可能先失败一次
- 随后权限弹窗才出现
- 用户确认后才可继续

这说明当前的底层 transport 已经可用，但：

- 权限请求时机
- UI 提示语义
- 授权后的自动续接

仍然值得优化。

更合理的方向是：

- 未授权设备显示“授权”而不是直接“连接”
- 授权未完成时不先报连接失败
- 授权成功后自动刷新并继续连接

### 设备重新枚举后权限可能丢失

USB 设备重新枚举后，设备节点可能变化，导致之前的权限状态失效。

因此后续必须继续坚持：

- 授权后主动刷新设备列表
- 重新检查 `hasPermission(device)`
- 不要只依赖广播结果

## 二类：运行时语义与边界

### 结束会话的语义必须继续保持清晰

当前仍需要持续坚持的语义是：

- 结束会话
  清理 scrcpy socket、forward、server、decoder、monitor
- 保留健康 ADB
  让 TCP/IP 和 USB 连接在健康时仍可快速复用
- 只有 transport 失效
  才移除对应 ADB 连接

如果为了“看起来清理更彻底”而在用户结束会话时盲目断开 ADB，就会破坏：

- 保活
- 快速重连
- 现有 TCP/IP 语义一致性

### USB detach 与逻辑断开不能重新混用

后续任何改动，只要重新把下面两件事混成一类，都应该视为回退：

- 用户主动结束
- USB transport 物理失效

## 三类：TLS / pairing 后续项

如果继续推进 Wireless Debugging / TLS，当前最值得优先做的是：

1. 保持 peer-based / pin-based 判定，不再回退到端口号判断
2. 给 `peers.json` 增加 version migration、cleanup、damage recovery
3. 补 transport 级 rebinding 测试
4. 让 endpoint hint 更多映射到 peer identity，而不是长期依赖 IP
5. 继续保持 app 层不重新接管 packet-level pairing 或 transport-level TLS 逻辑

## 四类：非阻断但值得修的项

### 会话持久化仍有告警

当前仍有两类非阻断告警值得后续处理：

- 异步保存编码器列表失败
- 保存编解码器选择失败

当前判断：

- 不影响 USB 建链
- 不影响 scrcpy 成功
- 但会污染日志

因此应被视为：

- 不阻断主链路
- 但值得清理的日志与状态一致性问题

## 五类：中期开发项

结合当前开发计划和新文档体系，仍值得继续推进的中期项包括：

- 设备重连逻辑优化
- 视频/音频渲染优化
- 连接/断开异步执行
- 设备旋转宽高比适配
- 音频编解码支持继续补齐
- UI 继续对齐更一致的交互风格

## 六类：文档与工程项

还值得继续完善的工程项包括：

- 日志标签和信号文档继续细化
- 新 `docs/` 与代码入口说明进一步对齐
- 把真正过时的 `docs_old/` 内容逐步淘汰

## 后续推进顺序建议

如果下一轮要继续做，我建议顺序如下：

1. 权限与授权交互优化
2. TLS / peer metadata 健壮性
3. 非阻断日志污染清理
4. 重连与渲染体验优化

## 一句话总结

当前开放问题已经不再是“主链路能不能跑”，而是“体验、边界和健壮性是否足够稳定”，后续工作应围绕这三点继续收口。
