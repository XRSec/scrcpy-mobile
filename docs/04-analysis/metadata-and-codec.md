## Metadata 问题的本质

metadata 读取类问题通常不是“读到了错误协议”，而是“在错误时机或错误方式读取了本应严格按协议处理的数据”。

这类问题一旦发生，常见表现包括：

- 设备名乱码
- codec id 异常
- 分辨率异常

## 为什么容易出错

主要有两个原因。

### 时序问题

socket 连上以后，server 侧未必已经把完整 metadata 准备好。

如果客户端过早读取，就可能拿到：

- 部分正确数据
- 缓冲残留
- 不完整字段

### 读取语义问题

如果读取逻辑没有严格遵守字段长度和完整读取语义，就会导致后续解析全部错位。

当前 socket 建立逻辑有一个非常重要的技术约束：

- server 会按 `Video -> Audio -> Control` 的顺序串行 `accept`
- 客户端必须先把需要的 socket 全部连上
- dummy byte 只应从第一个 socket 读取

如果先连接一个 socket 就立刻开始读，可能会把 server 卡死在后续 `accept` 阶段。

## 修复思路

这类问题的修复方向通常有三点：

1. 确保在正确时机开始读取
2. 严格按协议长度读取
3. 读取失败时保留足够的上下文日志

当前实现中的关键处理点包括：

- socket 建立后先读 video socket 的 dummy byte
- dummy byte 使用重试和超时保护
- metadata 读取应采用完整读取语义，而不是假设一次 read 就拿到完整字段

## 编解码器问题的本质

编解码器问题通常不是单点错误，而是三个变量组合后的结果：

- 用户选择
- 设备能力
- 当前格式兼容性

如果只看其中任意一个维度，就容易做出错误判断。

另外还要额外考虑第四个变量：

- 当前传输形态和设备身份是否变化

## 编解码器缓存的作用

缓存的作用不是替代检测，而是减少重复成本。

它更像是：

- 已知设备能力的短期记忆

因此缓存必须满足两个前提：

- 设备身份可确认
- 失效策略清楚

对远端能力来说，最直接的校验条件是：

- `deviceSerial` 是否一致
- 编码器列表是否非空

## 选择逻辑的关键

正确的编解码器选择逻辑，应满足：

1. 用户显式选择优先
2. 用户没选的部分由系统补齐
3. 补齐结果必须与已有选择兼容

如果用户只选了编码器，系统就必须检查当前已选解码器是否同格式。

如果不兼容，就不能偷懒沿用旧值，而要重新检测或重选。

## 实际判断建议

遇到编解码器问题时，建议先问五个问题：

1. 当前是用户手选还是系统自动选
2. 当前设备能力是否已重新检测
3. 当前缓存是否可信
4. 编码器和解码器是否同格式
5. 问题发生在创建前、创建中还是运行中

## 关键代码落点

这一类问题通常涉及：

- `ConnectionSocketManager`
- metadata 读取逻辑
- 编解码器检测逻辑
- `ScrcpyOptions` 中的最终选择方法

## 一句话总结

metadata 问题属于协议时序和读取边界问题，编解码器问题属于用户偏好、设备能力和格式兼容三者之间的组合问题，两者都不能只靠表面现象判断。
