## 文档目的

这一篇整理的是日常开发层面的操作性规则：

- 新功能怎么落位
- 新通用能力怎么放
- 怎么判断依赖方向
- 测试和性能应该关注什么

## 新增功能时的判断顺序

新增一个功能时，建议先问四个问题：

1. 这是用户功能，还是技术实现
2. 它属于哪个 feature
3. 它是否会进入会话主链路
4. 它是否应该沉到底层基础设施

## 新功能模块的放置建议

### 如果是用户功能

优先考虑放在 `feature/<name>/`。

但这里的子目录是可选组织方式，不是固定模板。

默认先从少量文件开始组织。

如果一个功能只有少量文件，或者某个子目录只会增加跳转成本，就不要为了形式完整强行拆目录。

只有当某类代码已经形成稳定边界时，才考虑补出局部子目录，例如：

- `ui/`
- `viewmodel/`
- `presentation/`
- `data/`

### 如果是技术实现

优先考虑放在 `infrastructure/<domain>/`。

例如：

- 新 transport
- 新 pairing 细节
- 新解码器技术实现

### 如果是稳定可复用基础能力

优先考虑放在 `core/`。

例如：

- 常量
- 文本对象
- 通用模型
- 公共 UI 原语

## 命名建议

命名应直接表达职责。

典型约定包括：

- `*Screen`
- `*Dialog`
- `*ViewModel`
- 真实业务动作名，例如 `startSession`、`loadDevices`

需要避免：

- 文件名与实际内容明显不符
- 目录里出现不表达增量信息的重复层级
- 用模糊技术名词给纯转发层命名
- 用 `Repository`、`Manager`、`Facade` 包装没有独立语义的中转逻辑

## 数据流建议

推荐的数据流仍然是：

- UI
- ViewModel / feature 局部编排
- runtime orchestration 或 infrastructure
- system / transport

重点不是机械遵循某种架构口号，而是：

- 数据流向尽量单向
- UI 不直接承担底层编排
- 运行时主链路不要在多个 feature 之间漂移
- 不自动长出 `ViewModel -> UseCase -> Repository -> Manager` 全套中间层
- 没有独立价值的 `use case` 或 `repository` 不要硬拆出来

## 测试建议

### 单元测试

优先覆盖：

- `core` 中的工具、模型、文本或规则逻辑
- `infrastructure` 中相对独立的协议与解析逻辑
- `feature` 中的 ViewModel 和状态转换

### 集成测试

优先覆盖：

- 会话启动和停止主链路
- ADB verify / forward / server 启动
- transport 失效后的恢复路径

### 回归测试思路

涉及 USB、TLS、decoder、socket 的改动，最值得做的是“主链路回归验证”，而不是只测局部函数。

## 性能约束

### 性能不只是帧率

当前项目里，性能问题至少分成：

- transport / socket
- decoder / render
- UI / interaction
- 日志与监控噪音

### 优先级判断

遇到性能问题时，先判断是：

- 真正的传输瓶颈
- decoder 压力
- UI 渲染压力
- 日志过多导致的观测噪音

不要把所有问题都直接归因于“buffer 不够大”。

## 文件规模与拆分

继续遵守：

- 常规目标：`500` 行以内
- 强约束：不要超过 `800` 行

当文件超限时，优先按职责拆：

- UI 与状态拆分
- coordinator 与技术实现拆分
- handler 按事件族拆分

如果文件不长，但需要跨多个 30 到 50 行小文件才能看懂主流程，应优先考虑合并，而不是继续细拆。

当前项目如果出现“文件很多但信息密度很低”的情况，应优先做去抽象化，而不是继续做目录美化。

## 评审时的快速检查项

1. 新代码放错层了吗
2. 新状态是否重复表达了已有事实
3. 新入口是否绕开了既有统一入口
4. 新日志是否让排障更清楚，而不是更吵
5. 新类名和目录名是否表达了真实职责

## 一句话总结

开发和测试层面最重要的不是“多写几个模块”，而是确保每次新增代码都沿着同一套边界、目录和主链路继续收敛，而不是重新长出一套旁路。
