## 目标

工程规则的目的，是保证项目在持续演进过程中仍然保持：

- 可读性
- 一致性
- 可维护性

## 国际化规则

项目采用代码化双语管理。

实践重点：

- 文本按模块归类
- 弹窗和主页面的取文方式要区分
- 不把文案散落到 UI 代码里

当前文本对象已经按模块拆分，例如：

- `CommonTexts`
- `SessionTexts`
- `SettingsTexts`
- `AdbTexts`
- `RemoteTexts`
- `CodecTexts`
- `LogTexts`

## UI 规则

项目 UI 风格明确，不应混入大量默认 Material 风格实现。

原则：

- 标题栏统一
- 输入框统一
- 开关统一
- 列表项统一
- 分组标题统一
- 分割线统一

开发时优先使用项目内现成组件，而不是直接落回默认 Material 组件。

这样做的目的，是减少不同页面之间的风格和交互离散度。

## 代码组织规则

### 常量集中

常量不应四处散落。

### 模型集中

同一模块的模型、枚举和接口应尽量稳定归类。

### API 兼容集中处理

系统版本兼容逻辑应统一封装，避免业务代码到处直接判断版本。

当前最重要的两个组织习惯是：

- `XxxModels.kt`
  集中同模块的数据类、枚举和接口。
- `Constants.kt`
  统一维护常量来源。

## 会话相关规则

项目里最重要的状态边界是会话。

因此需要坚持：

1. 配置态和运行态分离
2. 设备能力和用户偏好分离
3. 不轻易新增全局入口

如果确实需要全局访问器，应把它视为过渡能力，而不是新的默认调用方式。

## 代码评审规则

评审时应重点看：

1. 是否引入了新的硬编码
2. 是否破坏了既有边界
3. 是否让职责变得更模糊
4. 是否新增长调用链或空心包装层
5. 是否继续放大了单文件规模
6. 是否偏离了统一 UI 和文本规范

## 一句话总结

工程规则不是附加文档，而是保证项目不会在多人协作和多轮重构中逐渐失控的底层约束。
