## 文档目的

这一篇不是通用 Android 架构教程，而是约束后续 AI 在当前项目里如何做重构。

目标只有四个：

- 缩短主链路阅读路径
- 减少空心包装层
- 保持会话和运行时边界清楚
- 让单人维护场景下的改动成本更低

当前项目的默认立场应明确为：

- 先合并，再证明有必要拆
- 先压平，再决定是否抽象
- 宁可保留一个完整的大文件，也不要制造一串空心小文件

如果通用“最佳实践”和本文档冲突，以本文档为准。

## 适用范围

本文档适用于：

- `scrcpy-mobile/app`
- `app/src/main/java/com/mobile/scrcpy/android/`

当前顶层边界已经比较明确：

- `app`
  应用入口和顶层装配
- `core`
  稳定共享能力
- `feature`
  用户功能和界面
- `infrastructure`
  ADB、scrcpy、媒体和系统实现
- `service`
  前台服务和系统级生命周期

后续重构的重点不是再造一套新层次，而是把这几层继续收紧。

## 当前项目的现实约束

这个项目首先是单人主导维护，其次才是未来可能协作。

因此后续 AI 重构必须优先服务：

- 3 个月后的自己能快速找到改动点
- 新人能顺着一个功能一路看到底
- 会话主链路不被大量技术包装切碎

当前代码里已经存在一些细分子目录，例如：

- `ui`
- `viewmodel`
- `presentation`
- `model`
- `data`
- `component`
- `util`

这些目录不是必须继续扩张的模板，只能在确实增加信息密度时保留。

这也是当前重构最需要防止的方向：

- 继续把一个功能切成更多 30 行左右的小文件
- 继续增加只有一层转发意义的目录或类
- 把一个完整业务动作拆成多层技术名词包装

## 核心原则

### 先优化调用链，再优化文件行数

单个文件偏大不是第一问题。

更重要的是：

- 改一个功能是否要跳很多层
- 看一个问题是否要点开很多 30 行小文件
- 中间层是否真的提供了新的业务语义

在当前项目里，`10` 个 30 行中转文件，通常比 `1` 个 300 行但职责完整的文件更难维护。

### 一起变化的代码放一起

如果两段代码总是一起改，就不要为了“分层整齐”强行拆开。

拆分依据应是变化原因，而不是文件长度。

### 调用链尽量控制在 2 到 3 跳

从 `Screen` 到真实业务动作、会话运行时或底层能力，尽量不要超过 2 到 3 跳。

如果出现下面这种结构：

- `A -> B -> C -> D`

其中 `B`、`C` 主要只是转发、拼接或改名，应优先压平。

### 允许局部重复，禁止假抽象

一个逻辑没有形成稳定复用之前，允许在局部重复。

不要为了“以后可能会复用”提前引入：

- `UseCase`
- `Repository`
- `Manager`
- `Facade`
- `Wrapper`
- `Bridge`

### 抽象必须产生真实收益

只有满足以下至少一项时，才允许新增一层抽象：

- 隐藏了明显复杂度
- 隔离了副作用
- 有多个稳定调用方
- 有独立业务语义
- 明显提升测试价值
- 让调用方更容易理解

如果做不到以上任一条，就不要抽。

## 目录与职责规则

### `app/`

只放：

- `Activity`
- 顶层导航
- 顶层应用装配

不要把具体功能编排、协议细节或会话细节继续塞回 `app/`。

### `feature/`

`feature` 是当前项目的主要功能落点。

适合放在 `feature/<name>/` 的内容包括：

- `Screen`
- `Dialog`
- `ViewModel`
- 屏幕局部状态
- 屏幕局部 mapper
- 只服务该功能的模型和纯逻辑

`feature` 下可以有 `ui`、`viewmodel`、`presentation`、`model`、`data` 等子目录，但它们都不是强制模板。

只有在确实带来信息增量时才保留这些子目录。

默认情况下，后续 AI 不应再给 feature 增加新的子层。

如果一个 feature 现在已经拆得过细，优先考虑合并，而不是在原有细分目录之间继续搬运代码。

以下情况应优先就近放置，而不是继续拆层：

- 只被一个 `Screen` 使用的 UI 组件
- 只被一个 `ViewModel` 使用的模型
- 只为一个功能服务的数据转换
- 只在当前功能里出现的状态定义

### `infrastructure/`

这里只放技术实现边界，例如：

- `adb`
- `scrcpy`
- `media`
- 系统 API 或协议交互

`infrastructure` 的职责是承接技术细节，不是再包一层更空泛的技术名词。

在这一层重构时，优先按真实子域收敛：

- 连接
- 协议
- socket
- decoder
- transport

不要继续新增低信息量中间层。

### `service/`

只放前台服务及其直接生命周期协作对象。

不要把普通业务逻辑因为“需要后台跑”就顺手放进 `service/`。

### `core/`

这里只放稳定共享能力：

- 设计系统
- 国际化文本
- 常量
- 通用模型
- 通用存储
- 经过验证的基础工具

禁止把下列内容塞进 `core/`：

- 只被一个 feature 使用的逻辑
- 仍在快速变化的 feature 规则
- 屏幕专属 mapper
- 为了“以后可能复用”先搬走的代码

## Android 项目专用规则

### Compose UI 规则

`Screen`、`Dialog`、`Content`、局部 `Component` 优先放在 feature 内。

只有当 UI 原语已经成为稳定共享组件时，才进入 `core/designsystem/`。

如果一个组件只服务一个屏幕，就不要提到全局组件目录。

### ViewModel 规则

`ViewModel` 负责：

- 暴露屏幕状态
- 接收用户意图
- 组织当前功能的调用顺序

`ViewModel` 不应自动下沉出一整套：

- `UseCase`
- `Repository`
- `Manager`
- `DataSource`

只有在逻辑边界真的独立时，才允许继续分解。

### `presentation` / `viewmodel` 规则

当前项目里，部分 feature 使用 `viewmodel/`，部分 feature 使用 `presentation/`。

后续 AI 重构时应遵守：

- 延续所在 feature 既有风格
- 不为了统一目录名发起大规模搬家
- 不为了“更像标准架构”新增平行层

如果一个 feature 已有 `ViewModel` 作为主编排入口，就不要再平行引入一层 `presentation` 只做转发。

如果一个 feature 已经有 `presentation` 目录，但里面的类主要只是桥接、改名或中转参数，应优先评估是否并回 `ViewModel` 或 feature 局部逻辑文件。

### `data` / `model` / `util` 规则

这些目录只能在边界明确时存在。

禁止以下情况：

- `data/` 里只有一两个转发文件
- `model/` 里只有当前文件自己会用的数据类
- `util/` 里混入实际业务规则
- `common util` 变成无边界杂物间

更稳妥的做法是：

- feature 局部模型就近定义
- 纯规则逻辑放在 feature 局部文件
- 只有稳定复用后再抽到 `core/`

默认不新增 `data/`、`model/`、`util/` 子目录，除非已经出现清晰边界和持续收益。

## 当前项目的具体收敛方向

### `feature/remote`

这里是远控主链路最敏感的用户入口之一。

后续重构重点应是：

- 保持 `Screen -> ViewModel/Presentation -> Session/Infrastructure` 路径清楚
- 不把远控特有的编排逻辑随手抽进 `core/`
- 不新增只做转发的远控包装层

### `feature/session`

这里直接贴近会话管理。

后续重构重点应是：

- 会话列表、会话管理、分组管理各自边界清楚
- 局部 UI 状态不要扩散成全局状态
- 不重复制造新的会话入口

### `feature/device` / `feature/settings` / `feature/codec`

这些目录允许继续做功能内聚，但不应为了目录整齐继续切碎。

如果一个屏幕相关逻辑只在当前 feature 内使用，应优先留在 feature 内。

### `infrastructure/adb` / `infrastructure/scrcpy` / `infrastructure/media`

这三块已经是明确技术边界。

后续 AI 应优先在各自边界内部做整理，例如：

- 合并纯转发类
- 收敛协议与连接相关对象
- 让 decoder 链路更直

不要再在它们上面套一层新的“总管理器”。

## 拆分与合并标准

### 允许拆分

满足以下任一条件时，可以拆分：

- 有独立业务语义
- 有独立副作用边界
- 有多个稳定调用方
- 适合作为独立测试单元
- 拆出去后调用方明显更简单

### 必须避免的拆分

以下情况不要拆：

- 只是为了让文件变短
- 只是为了“分层更标准”
- 只是为了预留未来扩展
- 只是为了把一个完整流程切成多个技术名词文件

### 必须考虑合并的信号

出现以下情况时，应优先考虑合并：

- 文件只有 30 到 50 行且大多只是转发
- 只有一个调用方
- 上下层几乎总是一起修改
- 不点开下一层就看不到真实逻辑
- 文件名很抽象，但内容很薄
- 一个动作需要连续点开 3 个以上小文件才能看懂

## AI 执行顺序

后续 AI 在重构时，按以下顺序执行：

1. 先确认当前功能入口和真实主链路
2. 标出所有纯转发和低信息量包装层
3. 优先压平调用链
4. 再提取真正独立的纯逻辑或副作用边界
5. 保持行为不变
6. 一次只重构一个 feature 或一段明确链路

禁止一上来做：

- 全目录统一重命名
- 全 feature 统一子目录模板
- 一次性引入新的大分层

## AI 禁止事项

后续 AI 不得：

- 机械引入 `usecase/repository/manager/facade`
- 因为“未来可能多人协作”就提前堆层
- 把单一调用方代码抽成全局共享
- 把主链路拆成更多技术名词目录
- 发起没有阅读路径收益的大规模重命名
- 用 `Common`、`Base`、`Helper` 之类模糊命名替代真实职责

## 提交前自检

每次 AI 重构结束前，至少自检以下问题：

1. 功能入口是否比之前更清楚
2. 主链路是否更短
3. 包装层是否减少了
4. 是否减少了跨文件跳转
5. 是否没有新增假抽象
6. 是否没有把单一 feature 逻辑错误提到 `core`
7. 是否没有改变原有行为

如果以上问题大多不能回答“是”，这次重构大概率方向不对。

## 一句话总结

当前项目后续重构的关键，不是继续把 Android 分层做得更像教科书，而是让 `Screen`、会话、ADB、scrcpy、decoder 这些真实链路更短、更直、更好找。
