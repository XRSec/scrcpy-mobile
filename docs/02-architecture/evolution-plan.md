## 文档目的

这一篇不描述“现在是什么”，而描述“后续应该往哪里收敛、分几步收敛”。

它的作用是替代旧文档中分散的：

- 目标架构蓝图
- 目录结构重构方案
- 部分中长期迁移原则

## 当前为什么还需要继续演进

虽然项目已经完成大量结构收敛，但当前仍存在三类长期问题：

1. 运行时事实源仍然偏多
2. 目录结构还没有完全表达主链路
3. 一些过渡入口和历史实现仍在主路径附近存在

因此，后续的重点不是再加一套新概念，而是把现有收敛方向继续做实。

当前还需要特别警惕一个现实问题：

- 局部目录和中间层已经拆得偏细
- 一些文件的信息增量偏低
- 部分功能需要跨多个小文件才能追到真实逻辑

所以后续演进不只是“继续分类”，还包括“主动去掉没有收益的分类”。

## 目标态

理想的目标态可以概括为：

- 一条清晰的会话运行时主链路
- 一套稳定的状态、事件和资源边界
- 一个能从目录结构直接看懂的代码组织方式

## 目标分层

后续应继续朝下面这个逻辑收敛：

- `app`
  应用启动和顶层装配
- `feature`
  用户功能和界面
- `runtime`
  会话运行时编排、状态和投影
- `infrastructure`
  ADB、scrcpy、媒体、系统桥接
- `core`
  通用基础能力

当前项目还没有完整显式拆出 `runtime/` 顶层，但运行时相关代码已经在局部显性化。

## 目标运行时模型

后续的运行时更适合围绕这几个概念稳定下来：

- `SessionRuntime`
- `SessionCommand`
- `SessionEvent`
- `SessionState`
- `SessionResourceRegistry`
- read model / telemetry projector

其中最关键的原则是：

- 命令表达“想做什么”
- 事件表达“已经发生了什么”
- 状态表达“当前聚合结果是什么”
- 资源对象不直接塞进状态树

## 目标目录原则

目录结构继续收敛时，至少应满足：

1. 文件名表达真实职责
2. UI 目录只放 UI
3. 运行时编排和技术实现分离
4. 历史目录和中间层级不再长期保留
5. 主链路代码尽量按协作闭包聚合

## 分阶段迁移建议

### Phase 1：继续收口事实源

目标：

- 减少同一事实在多套状态系统里重复表达

重点：

- 继续明确 `SessionEvent` 的主事实角色
- 让监控和日志退回投影层
- 减少“字符串日志驱动状态”的情况

### Phase 2：继续收口运行时入口

目标：

- 减少底层随手拿当前会话的路径

重点：

- 让更多基础设施通过显式上下文工作
- 压缩全局访问器的使用范围

### Phase 3：继续做目录治理

目标：

- 让一条主链路需要跳转的目录变少

重点：

- 继续把运行时相关对象收拢
- 压缩 feature 内低信息量子目录
- 只在确有收益时保留 `presentation` 一类中间层
- 避免再出现信息增量很低的中间目录

### Phase 4：完善 read model 与 diagnostics

目标：

- 让 UI 和调试展示更多依赖投影模型，而不是直接读取原始内部状态

重点：

- `ConnectionProgressModel`
- `DiagnosticsModel`
- telemetry / monitor 的稳定投影

## 当前已经对齐的方向

从现有代码和新文档看，已经对齐的方向包括：

- 会话已成为运行时最小边界
- `SessionManager` / `Session` 已取代旧的混合入口
- `SessionState` / `SessionEvent` / component snapshot 已初步成型
- `SessionContext` 已显式进入连接和部分媒体链路
- USB 与 TCP/TLS 的差异已经进入专门处理逻辑

## 当前不建议做的事情

### 不建议重新发明一套新的大而全事务框架

当前真正的问题不是“缺框架”，而是要把已有边界继续收紧。

### 不建议为了目录整齐做空转重命名

如果重命名不能直接缩短主链路阅读路径，就不是优先级最高的改动。

### 不建议继续把已经过细的 feature 再拆小

当前很多问题不是“大文件太多”，而是“小文件太多但信息密度不高”。

如果一个功能已经需要连续跳很多层，优先做合并和压平，而不是继续增加子目录。

### 不建议让监控重新侵入业务主线

监控是投影，不是业务真相。

## 文件规模约束

后续继续遵守：

- 常规目标：单文件尽量控制在 `500` 行以内
- 强约束：单文件不要超过 `800` 行

一旦超限，应优先拆职责，而不是继续累计逻辑。

## 迁移时的判断顺序

遇到某个模块要重构时，建议先问：

1. 这是运行时编排，还是技术实现
2. 这段代码是否表达主链路上的关键动作
3. 这段状态是否已经被别处表达过
4. 这个目录是否真的帮助理解主线

## 一句话总结

后续演进的核心，不是再设计一套更大的架构图，而是持续把运行时事实、目录结构和主链路阅读成本收敛到更稳定的形态。
