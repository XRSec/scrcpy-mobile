# 架构原则

## 设计目标

项目的架构目标可以概括为一句话：

让代码结构服务于远控主链路，而不是让主链路被结构本身切碎。

## 分层原则

项目采用模块化分层，但分层不是为了形式整齐，而是为了让职责和依赖更清楚。

当前项目需要特别避免的，不是“层数不够”，而是“已经有层以后还继续细切”，最终把主链路切成大量低信息量文件和中间目录。

### core

负责稳定基础能力，不依赖上层业务。

### infrastructure

负责技术实现，把 ADB、scrcpy、媒体和底层连接细节封装起来。

### feature

负责业务功能和用户交互，把具体能力组织成可使用的功能模块。

### service

负责系统级生命周期、前台服务和保活。

### app

负责应用入口、导航和最终装配。

## 会话优先原则

项目的最小运行时边界是会话。

所有关键状态最终都应围绕会话理解：

- 当前配置
- 当前设备
- 当前连接过程
- 当前组件状态
- 当前异常

只要脱离会话边界去堆全局状态，系统就会更难理解。

当前代码里的直接体现包括：

- `SessionManager`
  管理当前活跃会话。
- `Session`
  承载配置快照、资源引用、状态流和事件入口。
- `SessionRuntimeStore` / `SessionRuntimeLifecycle`
  负责会话运行时装配和生命周期。

## 唯一事实源原则

同一个事实不应在多套系统里并行表达。

例如：

- ADB 是否已连接
- socket 是否已断开
- decoder 是否已失败

这些都应有单一正式入口，再由日志、监控和统计去消费，而不是在多个地方各自产生一份近似状态。

## 监控是投影，不是主线

监控、日志和统计的职责是：

- 观察
- 记录
- 聚合
- 展示

它们不应成为业务真相本身。

如果监控系统同时承担状态机职责，系统就会变成两套主线并行。

## 目录必须表达职责

目录结构应帮助读者快速找到主链路，不应制造更多跳转成本。

因此需要遵守几个规则：

1. 文件名必须表达真实职责。
2. UI 目录只放 UI。
3. 技术实现和运行时编排要分清。
4. 历史残留目录要及时清理。
5. 主链路代码应尽量聚在一起。
6. 不为了“看起来更分层”继续增加低信息量中间目录。

## 配置与运行态分离

项目中的配置态和运行态不是一回事。

配置态回答：

- 用户想怎么连
- 设备能力是什么
- 偏好设置是什么

运行态回答：

- 当前正在发生什么
- 哪个组件已启动
- 哪一步失败
- 是否需要清理或重连

只有把这两类信息分清，会话模型才稳定。

当前代码里，这种分离主要通过以下对象表达：

- 配置态
  - `ScrcpyOptions`
  - `SessionStorage`
- 运行态
  - `Session`
  - `SessionState`
  - `SessionEvent`
  - `SessionComponentStateSnapshot`

## 文件规模也是架构约束

文件过大不只是可读性问题，还会放大职责混乱。

因此拆分文件不是单纯的格式整理，而是架构治理的一部分。

但反过来，拆分过度同样是架构问题。

如果拆分只是制造更多 30 行左右的转发文件，复杂度并没有下降，只是被移动到了更多路径上。

## 关键边界的技术表达

### 会话边界

会话边界至少要统一以下归属：

- `sessionId`
- `deviceIdentifier`
- `adbConnection`
- codec 检测结果
- monitor bus
- session state
- component snapshot

### 连接边界

连接边界至少要区分：

- TCP/TLS 网络连接
- USB transport 连接
- 逻辑会话结束
- 物理 transport 失效

### 监控边界

监控边界至少要区分：

- 业务主状态
- 日志投影
- 统计投影
- 调试辅助事件

## 一句话总结

这套架构原则的核心，不是追求抽象层次更多，而是通过会话边界、唯一事实源和职责清晰的目录，让远控主链路更容易被理解和维护。
